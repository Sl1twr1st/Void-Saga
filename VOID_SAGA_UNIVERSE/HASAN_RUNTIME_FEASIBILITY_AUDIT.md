## STEP 3 — Runtime Type Decision

### A) Full Runtime Candidate (v1.0.0, 16 V2 sections)

**Rejected.** 3 sections unsupported, 7 sections partial. Would require heavy [I] marking across more than half the runtime. A v1.0.0 with more [I] than [E] violates V2 tag discipline requirements.

### B) Compact Runtime Candidate (v0.5.0, reduced sections)

**Feasible.** A reduced-section format could document:
- Identity + Core Wound (Split Self)
- Vesica Piscis interface (provider + participant)
- Merge residue
- Relationship contracts (5 targets)
- Removal event and vacancy created
- Open questions

This would be ~8 sections — the SUPPORTED + PARTIAL sections, omitting the 3 UNSUPPORTED. It would be explicitly marked v0.5.0 with a note that full V2 compliance is deferred pending canon expansion.

### C) Deferred Runtime Candidate

**Reasonable alternative.** Hasan's absence from Timer 1800–2500 means his canon arc is incomplete. If Jali writes Timer content that shows Hasan post-removal, the evidence profile changes. Deferring avoids creating a runtime that may need immediate revision.

However: Hasan is referenced by 3 existing runtimes (Julia, Delphie, Sevraya). His Merge contract is one-sided from all three. A compact runtime would close those contracts even if the runtime itself is partial.

## STEP 4 — Core Six Kunci Analysis

### Is Hasan one of the Six Kunci?

**No.** The Six Kunci are: Julia, Delphie, Sevraya, Gwaneum, Agnia, NiuNiu. (Timer 1800, VOID_ONTOLOGY.md)

Hasan is explicitly excluded. He is absent from Timer 1800–2500. He has no sigil among the six canon sigils. His seventh sigil (⟲⟁⟲) is [PC] — unconfirmed by Codex or chapter narrative.

### Is Hasan structurally required for Core Runtime Topology?

**No.** The five-runtime core set (NiuNiu, Sevraya, Julia, Delphie, Agnia) forms a stable topology without Hasan. The Merge contract is asymmetric (Julia and Delphie have runtimes; Hasan does not) but the asymmetry does not break the topology.

### Is Hasan required for Runtime Expansion Topology?

**Partially.** Three existing runtimes reference him. His Merge contract is one-sided. A compact runtime would:
- Close the Merge contract (Julia↔Hasan, Delphie↔Hasan become symmetrical)
- Document the vacancy that Gwaneum fills (Enam Simpangan Air slot)
- Provide the Vesica Piscis provider interface that VOID_ENTRY_PROTOCOL references

These are structural gaps, not critical gaps. The topology functions without Hasan. It functions better with him.

## STEP 5 — Recommendation

## CREATE_HASAN_V0_5

**Compact runtime.** Reduced sections. Explicitly marked v0.5.0. Defers full V2 compliance to canon expansion.

### Justification

1. **Evidence insufficient for v1.0.0.** Three V2 sections unsupported. Would violate tag discipline requirements.

2. **Evidence sufficient for compact runtime.** Core wound, invocation, costs, relationships, and removal event are all documented. Eight sections can be populated without invention.

3. **Deferral cost is real but acceptable.** Three existing runtimes have asymmetric Hasan contracts. A compact runtime closes them. Deferral leaves them open indefinitely.

4. **Seventh sigil [PC] is the largest variable.** If confirmed by future canon, Hasan's runtime class shifts from "Formless" to "Witness." A v0.5.0 can be upgraded to v1.0.0 when that happens. A deferred runtime cannot.

### Compact Runtime Sections (proposed)

```
1. Runtime Status (v0.5.0 declaration)
2. Identity
3. Core Wound (Split Self)
4. Invocation (Vesica Piscis provider + participant)
5. Cost Pattern
6. Residue Pattern
7. Relationship Interfaces
8. Removal Event (taken by Zero, vacancy created)
9. Canon Boundaries
10. Runtime Summary
```

Omits: Runtime Class, Primary Contradiction, Defense System, Consequence Pattern, Protocol Interfaces, Evolution Stages, Failure Mode, Terminal State — all unsupported or too thin for even compact documentation.
