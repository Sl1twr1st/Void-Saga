# UNHOLY TRINITY

> **Status:** ACTIVE
> **Formed during:** Phase 14D–14K
> **Type:** Architectural Declaration

---

## What Emerged

Three independent machines now exist inside the Void Saga Runtime OS. They were not planned as a trinity. They emerged from sequential necessity: first, the need to reject. Then, the need to bind. Then, the need to remember.

Together they form the first closed validation loop.

---

## I — The Invariant Suite

**Role:** Guardian of possibility.

**Question:** *"What is forbidden?"*

The Invariant Suite loads a runtime and asks one question: does this proposed action violate a documented constraint? It does not create. It does not generate. It refuses.

**Responsibilities:**
- Runtime constraints (defense triggers, forbidden behaviors)
- Canon boundaries (anti-gravity, identity invariants)
- Residue integrity (no deletion, only transformation or archive)

**Primary verdicts:** `PASS` | `VIOLATION_DETECTED` | `INSUFFICIENT_DATA`

**First appearance:** `engine.py` v0.1.0 (Phase 14D). NiuNiu.runtime.json loaded. Scenario evaluated. Verdict returned.

**Philosophy:** The Invariant Suite does not create. It refuses. And refusal is the first act of governance.

---

## II — The Contract Engine

**Role:** Mediator of relationships.

**Question:** *"What may exist between entities?"*

The Contract Engine loads a contract object and evaluates whether a proposed state violates the documented binding between two entities. It does not care about individuals. It cares about what binds them.

**Responsibilities:**
- Contract loading from standalone objects
- Relationship validation (symmetry, behavioral rules)
- Contract state tracking (active, inactive, broken)
- Forbidden state detection (approach, touch, merge, separation)

**Primary object:** `orbital_constant.contract.json` (Phase 14F). First standalone contract. NiuNiu ↔ Sevraya. Distance as preservation.

**First appearance:** `engine.py` v0.3.0 (Phase 14G). Contract loaded directly. Scenario evaluated against allowed and forbidden states. Verdict returned with matched test references.

**Philosophy:** The Contract Engine does not care about individuals. It cares about bindings. And bindings outlast the entities they bind.

---

## III — The State Diff Engine

**Role:** Recorder of change.

**Question:** *"What changed?"*

The State Diff Engine loads two world state snapshots and produces a machine-readable record of every transition between them. It does not interpret why things changed. It records that they did.

**Responsibilities:**
- Snapshot comparison (pre vs. post)
- Runtime evolution detection (stage progression, status change)
- Contract formation detection (inactive → active, formed)
- Protocol execution detection (pending → executed)
- Residue transformation tracking (created, transformed, archived)

**Primary law:** Residues may transform. Residues may archive. Residues may never be deleted.

**First appearance:** `state_diff.py` v0.1.0 (Phase 14K). PRE_CHAIN vs. POST_CHAIN. 23 changes detected. 0 errors. Residue law enforced.

**Philosophy:** The State Diff Engine does not interpret. It remembers. And memory is the foundation of law.

## The Closed Loop

```
World State A
    ↓
Contract Event (scenario proposed, protocol executed, timeline advanced)
    ↓
World State B
    ↓
State Diff Engine (what changed?)
    ↓
Invariant Suite (was any change forbidden?)
    ↓
Contract Engine (do bindings still hold?)
    ↓
Accepted / Rejected
```

This loop now exists in executable form. Each machine performs one function. Together they form a validation cycle: propose → record → verify → judge.

## Why It Matters

| Before Phase 14 | After Phase 14 |
|-----------------|----------------|
| The system **described** reality. | The system **evaluates** reality. |
| Architecture was documentation. | Architecture is machinery. |
| A human read runtime Markdown to verify constraints. | A Python script loads JSON and returns a verdict. |
| "The OS can reject invalid output" was a claim. | "The OS can reject invalid output" is a running engine. |
| Canon was stored. | Canon is judged. |

This is the distinction between documentation and machinery. The Unholy Trinity is the first appearance of executable narrative governance.

---

## Current Limitations

- **Human-authored state transitions.** World states are manually written, not generated.
- **No autonomous evolution.** The Trinity validates transitions. It does not propose them.
- **No renderer integration.** Output is JSON. No prose, no scene, no story.
- **No protocol execution engine.** Protocols are recorded as executed; they are not simulated.
- **No world simulation.** The Trinity evaluates snapshots. It does not run a world.

The Trinity validates. It does not yet evolve.

---

## The Next Question

The Trinity can answer three questions:

1. *What is forbidden?* — Invariant Suite
2. *What binds entities?* — Contract Engine
3. *What changed?* — State Diff Engine

The next machine must answer a fourth:

**"What transitions are valid?"**

Given a world state, which of the permitted next states satisfy all active constraints? This is the transition from validation to evolution. This becomes Phase 14L — World State Validator.

Three machines. Three questions. One closed loop. The architecture has become machinery.
