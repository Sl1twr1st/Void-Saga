# Bab 00 Fork MVP — Genesis Node Forkability

> 2026-06-27. Decision log. Two build phases in one day: MVP core (morning) + v0.2 language bridge (afternoon).

## Trigger

One-month focus: "Bab 00 bisa bercabang tanpa kehilangan hukum."

Goal: build a minimal MVP proving Bab 00 can serve as a forkable genesis node. Not the full compiler. Not the whole novel. Just Bab 00 + rule-based checker + demo verdicts.

## Phase 1 — MVP Core (morning)

### What was built

| Artifact | File | Purpose |
|----------|------|---------|
| Genesis Node | `mvp/genesis/bab00.node.json` | Bab 00 metadata: 10 sub-chapters, 3 characters, 3 key mechanisms |
| Fork Points | `mvp/genesis/fork-points.json` | 3 natural fork points: fp-001 (00:03 Bug Janggal), fp-002 (00:07 Folder Error), fp-003 (00:08 Genesis Error) |
| Law Mapping | `mvp/laws/bab00-laws.json` | 7 Bab 00 laws with `block_on_violate` / `price_on_violate` flags |
| Fork Examples | `mvp/forks/fork-0{1,2,3}-*.json` | PASS (hesitation), PRICE_REQUIRED (rejection), BLOCKED (forced installation) |
| Fork Checker | `mvp/checker/check_fork.py` | Rule-based verdict engine. Reads fork + laws → PASS / PRICE_REQUIRED / BLOCKED / VALID_FORK |
| CLI Wrapper | `scripts/check-fork` | Entry point: `./scripts/check-fork path/to/fork.json` |
| README | `mvp/README.md` | Demo instructions, structure, how to create forks |

### Design decisions

1. **Bab 00 only.** The 7 laws are specific to Bab 00. No generalization attempted.
2. **Rule-based, not LLM.** The checker evaluates declared violations against law flags. No prose understanding. No API calls. This is honest about current capability.
3. **Violations declared by fork author, not auto-extracted.** The MVP proves the verdict mechanism, not automated extraction. Extraction is Fase 2.
4. **7 laws, not all of World DNA.** Narrow scope. Each law has evidence citations from Bab 00 prose.
5. **Verdict logic:** block_on_violate → BLOCKED; price_on_violate → PRICE_REQUIRED; no violations + trivial change → PASS; no violations + meaningful divergence → VALID_FORK.

### Verdict exit codes

PASS/VALID_FORK → 0, BLOCKED → 2, PRICE_REQUIRED → 3.

## Phase 2 — v0.2 Language Bridge (afternoon)

### Trigger

Existing fork examples were English-dominant. The novel and target users are Indonesian. The checker shouldn't pretend to be multilingual yet — but Indonesian users need to read fork premises in their language.

### Design

```
User-facing (🇮🇩):   title_id, premise_id, change_id, immediate_effect_id
                    → Displayed in CLI output. Read by users.

Internal bridge (🌐): internal_summary_en
                    → Used by checker for internal evaluation. English.
                    → Temporary. To be removed when checker is native Indonesian.

Backward compat:     title, premise, change (original English fields)
                    → Remain as fallback when Indonesian fields absent.
```

### What was added

- 4 Indonesian fields on all existing fork records + 1 new Indonesian fork
- `internal_summary_en` bridge field on all forks
- Checker v0.2: prefers `internal_summary_en` internally, displays Indonesian surface in output, shows bridge note
- `fork-04-valid-id.json`: full Indonesian VALID_FORK example (Printer Note)
- README rewritten bilingual with bridge explanation

### Explicit constraints (not done)

- No embeddings. No LLM calls. No AI generation.
- No broad law model changes.
- No expansion beyond Bab 00.
- Bridge is declarative — fork author fills both sides.
- Checker does not understand Indonesian. The bridge is honest about this.
