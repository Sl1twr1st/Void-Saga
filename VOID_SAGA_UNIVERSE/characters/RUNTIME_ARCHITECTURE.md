This document defines the identity-compression architecture for character runtimes. It is not a Phase 11 Protocol because it does not describe an in-world ritual that can fail.

# RUNTIME ARCHITECTURE

Runtime adalah behavioral DNA karakter atau entitas.

Tujuannya bukan menjawab "siapa karakter ini?" secara trivia, tapi menjawab:

> Jika karakter ini dijalankan sebagai proses dalam scene, chat, game, app, atau fork, pola apa yang membuatnya tetap dirinya?

## Runtime Stack

1. **Core Wound**
   Luka utama yang membentuk perilaku.

2. **Defense Pattern**
   Mekanisme default untuk menghindari, mengendalikan, atau bertahan dari luka.

3. **Trigger**
   Input yang mengaktifkan defense.

4. **Expected Reaction**
   Output default saat defense aktif.

5. **Recorded Anomaly**
   Momen canon ketika karakter melanggar pattern.

6. **Revealed Truth**
   Kebenaran yang muncul karena anomaly.

7. **Somatic Signature**
   Tubuh, diam, gesture, interface, atau system presence sebelum bahasa muncul.

8. **Voice Grammar**
   Cara bicara, menolak bicara, memotong kalimat, atau memakai sistem sebagai bahasa.

9. **Canon Boundary**
   Apa yang diketahui, tidak diketahui, disembunyikan, disangkal, atau belum bisa diakui.

10. **Fork Logic**
    Cara karakter boleh berubah di alternate release tanpa kehilangan DNA.

## Runtime Laws

- Karakter tidak boleh menjelaskan dirinya terlalu mudah.
- Voice bukan aksen. Voice adalah pola luka yang sudah belajar grammar.
- Anomaly lebih penting dari trait.
- Chat interface harus menghormati canon boundary.
- Fork boleh mengubah kejadian, tapi harus membayar konsekuensi psikologisnya.
- Karakter tidak boleh dibuat "lebih sehat" tanpa proses.
- Karakter tidak boleh dibuat "lebih gelap" tanpa alasan struktural.

## Runtime Test

Sebuah runtime dianggap valid kalau karakter tetap recognizable dalam 5 kondisi:
- Dipuji.
- Ditinggalkan.
- Dibohongi.
- Ditawari kuasa.
- Diminta mengakui takut.
