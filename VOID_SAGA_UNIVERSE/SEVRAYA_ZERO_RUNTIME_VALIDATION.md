## 4. Architecture Weaknesses Exposed by Dual Identity

### Weakness 1: No "Shared Body" Convention in Architecture

**Problem:** The architecture has no formal mechanism for two runtimes sharing one body. Sevraya and Zero share somatic signature, evolution stages, and physical location, but the architecture assumes one runtime = one body.

**Current workaround:** "See also" links + explicit shared-element notes + non-duplication discipline. This worked for Sevraya/Zero but required conscious effort.

**Recommendation:** Add a "Shared Body Protocol" to RUNTIME_PROTOCOL.md for future dual-identity characters (if any exist — Sevraya/Zero may be unique in Void Saga canon).

**Severity:** LOW — may never recur. The workaround is adequate for a single case.

---

### Weakness 2: [E]/[I] Tagging Uneven Across Runtimes

**Problem:** NiuNiu.runtime.md (written before the [E]/[I] convention was formalized) does not use [E]/[I] tags. Sevraya and Zero runtimes do. This creates inconsistent metadata across the runtime library.

**Current state:** NiuNiu needs retroactive [E]/[I] tagging. The convention was added to the architecture AFTER NiuNiu was written. This is expected for a reference implementation — the first runtime tests the architecture, the architecture improves, the first runtime needs updating.

**Recommendation:** After the validation cycle completes, retroactively tag NiuNiu.runtime.md with [E]/[I]. Low priority — the runtime is functionally correct; the tags improve auditability but don't change behavior.

**Severity:** LOW — cosmetic consistency issue.

---

### Weakness 3: Core Desire Inference Gap (Zero)

**Problem:** Zero's Core Desire section acknowledges that Zero may be "pre-desire" — a consciousness that exists before the category "want" was installed. The architecture assumes every character has a core desire. Zero may not.

**Current workaround:** "If forced to articulate: Zero wants to be acknowledged as real without being mistaken for Sevraya. [I]" — plus the caveat that the question itself may be inapplicable.

**Recommendation:** Add an architecture note: *For characters whose nature precludes desire (interfaces, processes, conditions), Core Desire may be replaced with Core Function — a statement of what the entity exists to do, not what it wants.* This covers Zero (interface), Ophiuchus (condition), and potentially Dorian Grey (process).

**Severity:** MEDIUM — will recur for non-person entities. Architecture should explicitly permit Core Function as alternative to Core Desire.

---

### Weakness 4: Trigger Conditions for Involuntary Switches

**Problem:** Sevraya's trigger "Zero activation" is marked INVOLUNTARY — she doesn't choose it, she yields. This is a new trigger type not accounted for in the architecture's intensity scale (LOW/MEDIUM/HIGH/CRITICAL).

**Current workaround:** INVOLUNTARY added as ad-hoc intensity level in Sevraya.runtime.md §5.

**Recommendation:** Add INVOLUNTARY to the intensity scale definition in RUNTIME_PROTOCOL.md: *INVOLUNTARY: Trigger activates automatically. Character does not choose the response. Character may observe but cannot prevent.*

**Severity:** LOW — one-line protocol update.

---

### Weakness 5: Canon Gravity for Cosmological Characters

**Problem:** Sevraya's Canon Gravity includes Era Hydrochoos stewardship — which is simultaneously character gravity AND world-level event. The architecture's forbidden dependency (World DNA must not reference runtimes) is respected, but the runtime's gravity claim references a cosmological outcome.

**Current handling:** The gravity is marked as character tendency ("She ends as Era steward"), not world declaration ("Era Hydrochoos continues because of her"). The distinction is maintained but the proximity is uncomfortable.

**Recommendation:** Accept this as inherent to characters with cosmological function. The architecture correctly separates character tendency from world rule. The proximity is not a flaw — it's accurate modeling of a character whose personal arc IS cosmological.

**Severity:** LOW — accepted boundary proximity.
